# 코드 품질 기준 (토스 프론트엔드 펀더멘탈)

코드 작성·리뷰의 판단 잣대. 4대 기준으로 스스로 검토한다.
출처: https://frontend-fundamentals.com/code-quality/code/

## 4대 기준

- **가독성** — 한 번에 읽히는가. 맥락이 과하게 얽혀 있지 않고, 한 함수가 한 추상화 수준만 다루는가.
- **예측 가능성** — 이름·시그니처·규약이 동작을 정확히 예고하는가. 같은 종류의 함수는 같은 형태(반환·에러·키)를 지키는가.
- **응집도** — 함께 바뀌는 것이 함께 있는가. 값의 단일 소스가 있는가(색·상수·타입을 여러 곳에 복제하지 않는가).
- **결합도** — 한 곳을 바꿀 때 파급이 좁은가. 하위 레이어가 상위를 모르는가(FSD 단방향).

## 핵심 트레이드오프 (가장 중요)

**성급한 추상화보다 중복을 허용한다.** 4대 기준은 서로 충돌한다 — 결합도를 낮추려면 중복을 감수해야 할 때가 있다. "응집도를 높인다"는 명분으로 아직 근거가 약한 공용화를 만들지 않는다.

- 판단 규칙: **중복이 3회 이상 · 함께 바뀜이 확실 · 형태가 진짜 같을 때만** 공용화한다. 2회뿐이거나 데이터가 서로 다르면 **중복을 둔다**.
- "책임은 하나씩" — 하나로 묶으면 여러 책임이 섞이는 추상화는 만들지 않는다.

## 이 코드베이스의 본보기

- **좋은 예(따를 것):**
  - `ROUTES` 경로 헬퍼 — 경로 문자열을 하드코딩하지 않고 여러 곳이 단일 소스를 참조(응집도).
  - `entities/poll/api/mappers.ts`의 `build*`/`map*` 재사용 — 라우트마다 매핑을 복붙하지 않음(응집도).
  - 뮤테이션 `onSuccess`가 무효화 Promise를 반환해 리페치 완료까지 `isPending` 유지 — 같은 규약을 모든 쓰기 훅이 공유(예측 가능성).
  - meta jsonb 안전 파서 5종의 기본값 방어 — 계약 밖 값이 화면을 깨지 않음(예측 가능성).
- **나쁜 예(하지 말 것):**
  - 4개 뮤테이션 훅을 하나의 factory로 묶기 — 무효화 키·Promise 반환·낙관적 업데이트 여부가 제각각 정당한데 묶으면 "책임 하나씩"·"중복 > 성급한 추상화" 위배.
  - 데이터가 다른 RPC 에러 맵을 억지 공용 헬퍼(`mapRpcError`)로 묶기 — 3번째 사용처가 생기면 그때.
  - 토큰과 동일한 hex 하드코딩(`#171717` 대신 `COLOR.ink`/토큰) — 색의 단일 소스 훼손(응집도).
  - 동작을 문자열 sentinel로 분기(`color === "var(--color-primary)"`) — 의미 기반 prop(`tone`)으로(예측 가능성).

## FSD와의 관계

결합도 기준은 FSD 단방향 의존(`shared ← entities ← features ← widgets ← views`)으로 구체화된다. 동일 레이어 간 import 금지, 배럴 공개 API 준수는 `architecture.md`를 따른다.
